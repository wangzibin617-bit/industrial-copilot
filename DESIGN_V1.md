## 1. 产品定位

### 1.1 目标客户 — 组织类型（Organization Type）

产品不是卖给"某个人"，而是卖给"一个组织"。不同组织类型有不同的协作结构和付费逻辑。

| 组织类型 | 典型画像 | 规模 | 核心协作链 | 付费决策者 |
|---------|---------|------|-----------|-----------|
| **工业自动化代理商** | 代理施耐德/信捷/汇川/西门子等品牌的经销商，年销售额3000万-5亿 | 30-300人 | 销售→售前→采购→商务→客户 | 销售总监/总经理 |
| **系统集成商（SI）** | 承接工厂自动化改造、产线集成的工程公司，用代理商的货做项目 | 20-100人 | 项目经理→技术负责人→采购 | 技术总监/老板 |
| **区域经销商** | 深耕单一区域的工控产品经销商，以走量为主，技术能力偏弱 | 10-50人 | 销售→厂商技术支持 | 老板（个体决策） |

**MVP聚焦：工业自动化代理商**（以下简称"代理商"），原因：
- 品牌多、产品线复杂，选型压力集中在售前
- 销售-售前-采购-管理层四级协作链路完整
- 知识沉淀需求最迫切（多品牌+多销售+高流动率）
- 付费意愿可由"提效ROI"论证

### 1.2 代理商的组织协作结构（真实还原）

```
代理商组织内部视角：

                    ┌──────────────┐
                    │  总经理/老板   │
                    │  (最终决策)    │
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
     ┌────────▼──────┐ ┌──▼──────┐ ┌──▼──────┐
     │  销售总监      │ │ 技术总监 │ │ 财务/商务│
     │  (管销售团队)   │ │(管售前团队)│ │         │
     └──────┬───────┘ └──┬──────┘ └──┬──────┘
            │            │            │
    ┌───────▼───────┐ ┌─▼──────────┐ │
    │ 销售/客户经理   │ │ 售前工程师   │ │
    │ (对接客户需求)  │ │(选型+方案+  │ │
    │               │ │ 技术应答)    │ │
    └───────────────┘ └─────────────┘ │
                            │         │
                    ┌───────▼─────────▼──┐
                    │   采购/商务         │
                    │   (询价+下单+货期)   │
                    └────────────────────┘
```

关键认知：
- **销售**和**售前**在组织内是**两个不同角色**，有明确的"需求→方案"转交关系
- 大项目方案需要**技术总监或销售总监审批**才能发给客户
- **采购/商务**决定最终报价——销售不知道真实成本
- **知识库管理员**通常是售前团队中的一人兼任，或独立的产品运营岗

### 1.3 真实业务痛点（按组织协作视角）

| 编号 | 痛点 | 涉及角色 | 发生场景 | AI可替代价值 |
|------|------|---------|---------|------------|
| P1 | **销售→售前传递损耗**：销售不懂技术，口头描述需求给售前，"客户说要控制水泵" → 售前追问3轮才搞清楚 | sales_rep → presales_engineer | 销售把客户需求转述给售前 | AI帮销售把模糊需求结构化 |
| P2 | **售前瓶颈**：一个售前同时对接5-8个销售，需求排期2天，销售干等 | presales_engineer | 售前排期积压 | AI抢跑完成80%选型工作，售前只审核 |
| P3 | **跨品牌对比主观**：销售推自己提成高的品牌；客户要跨品牌对比时无人能客观回答 | sales_rep / presales_engineer | 客户比价阶段 | AI无品牌偏好，客观对比 |
| P4 | **方案质量波动**：不同售前出的方案格式不同，好的方案靠人写，差的方案靠模板 | presales_engineer / sales_manager | 方案审核环节 | AI统一方案框架，降低标准差 |
| P5 | **经验不可复用**：老销售/老售前离职，客户偏好、历史方案、报价策略全丢 | 所有角色 | 人员流动 | 对话记录+方案沉淀→知识库永久留存 |
| P6 | **采购可行性未知**：售前选了型号，但采购发现没库存/货期长/价格不好，方案要重做 | presales_engineer → procurement | 方案→下单环节 | 后期扩展：采购反馈闭环 |

### 1.4 MVP 边界（按角色和阶段划分）

#### ✅ MVP 阶段（V1.0）— 聚焦 sales_rep + presales_engineer

| 模块 | 功能 | 使用角色 | 说明 |
|------|------|---------|------|
| AI Copilot | 自然语言选型、对比、RAG问答 | sales_rep, presales_engineer | 两个角色共用同一对话入口 |
| 产品库 | 产品浏览、搜索、参数查看 | sales_rep, presales_engineer | 平台预置200+型号 |
| 知识库 | 文档上传、RAG检索 | presales_engineer (上传), sales_rep (仅检索) | 上传权限限售前 |
| 对话记录 | 历史对话查看、搜索、复用 | sales_rep, presales_engineer | 组织内按角色可见 |
| 方案中心 | 方案生成、编辑、确认、导出 | presales_engineer (生成+确认), sales_rep (查看) | 方案需售前确认 |
| 对比中心 | 产品对比 | sales_rep, presales_engineer | 共享 |
| 团队管理 | 成员邀请、角色分配 | organization_admin | 基本角色管理 |
| 登录 | 邮箱+密码 | 所有用户 | Supabase Auth |

#### ❌ MVP 不做

- **采购/商务角色**（procurement）的功能（询价、货期、成本校验）
- **审批工作流**（sales_manager审批方案）
- **角色工作台**（按角色定制的Dashboard）
- 微信小程序
- 价格引擎
- 库存对接
- 客户CRM

#### 📋 后续版本路线

| 版本 | 核心扩展 | 新增角色 | 关键功能 |
|------|---------|---------|---------|
| V1.1 | 审批流程 | sales_manager | 方案审批→导出控制，团队数据看板 |
| V1.2 | 采购协同 | procurement | 采购可行性标记、货期反馈、成本区间 |
| V1.3 | 角色工作台 | 全部 | 按角色定制的Dashboard、消息通知 |
| V2.0 | 微信小程序 | sales_rep | 移动端选型、拍照识产品 |

### 1.5 商业价值

| 维度 | 量化价值 | 论证逻辑 |
|------|---------|---------|
| **销售→售前传递效率** | 从3轮沟通减到1轮 | AI帮销售结构化需求，售前拿到时已有初选结果 |
| **售前人均产能** | 从日处理8需求→15需求 | AI完成80%选型和方案初稿，售前聚焦审核 |
| **方案输出速度** | 从4h→30min | AI生成初稿+人工修订 |
| **新人上手** | 从3个月→2周 | 知识库+AI替代师傅带教 |
| **方案品质一致性** | 标准差降低 | AI统一框架，人工审核保底 |

**定价建议（SaaS）**：
- 基础版：¥299/月/组织（≤5人，AI调用300次/月）
- 专业版：¥999/月/组织（≤20人，AI调用1500次/月）
- 企业版：¥2999/月/组织（不限人数，AI调用5000次/月）

### 3.1 产品目标（可衡量）

| 目标 | 衡量指标 | 目标值（上线后3个月） | 验证方式 |

### 3.2 核心场景（按角色协作）

| S2: 销售→售前协同选型 | sales_rep | presales_engineer | 销售提交结构化需求 | 售前审核+修正选型 | AI Copilot + 对话共享 |

### 3.3 用户角色与权限体系（RBAC + Organization）

#### 3.3.1 组织层级

```
platform (平台层)
  │
  ├── super_admin (平台超管)
  │     └── 管理所有组织、系统配置、用量监控
  │
  └── organization (组织层，一个组织=一个代理商/SI/经销商)
        │
        ├── organization_admin (组织管理员)
        │     └── 管理本组织成员、角色分配、查看用量
        │
        ├── sales_manager (销售总监/业务负责人)
        │     └── 管理销售团队、审批大项目方案(后续)、查看团队数据
        │
        ├── sales_rep (销售/客户经理) ★ MVP核心
        │     └── 对接客户需求、自助选型查询、发起协同请求
        │
        ├── presales_engineer (技术/售前工程师) ★ MVP核心
        │     └── 选型审核、方案确认、技术问答、知识库贡献
        │
        ├── procurement (采购/商务) ◇ 后续版本
        │     └── 询价、货期反馈、成本校验
        │
        └── knowledge_admin (知识库管理员/产品运营)
              └── 维护产品数据、管理知识文档、方案模板
```

### 3.5 ToB 业务逻辑

| 业务规则 | 说明 | 实现方式 |
|---------|------|---------|
| BR1: 多租户隔离 | 不同组织数据完全隔离 | RLS + organization_id |
| BR2: 角色能力边界 | 不同角色看到的菜单、按钮、功能不同 | RBAC middleware + 前端条件渲染 |
| BR3: 销售→售前协同 | 销售可发起的对话，售前可介入并继续 | 对话所有权 + 组织内可见 |
| BR4: 方案确认职责 | 只有 presales_engineer 可标记方案为"已确认" | API层角色校验 |
| BR5: 知识上传权限 | knowledge_admin + presales_engineer 可上传 | API层角色校验 |
| BR6: 数据归属 | 对话/方案记录归属创建者user_id + 组织org_id | 双字段归属 |
| BR7: 试用期 | 新组织7天免费试用 | trial_ends_at |
| BR8: 审批流(V1.1) | 大项目方案需 sales_manager 审批后导出 | approvals表 + 状态机 |
| BR9: 离职交接 | 成员离开组织时，其数据保留在组织内，不可删除 | 软删除 + 数据归属组织 |

### 3.7 风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| R1: AI幻觉 | 推荐错误型号，方案不可用 | 多层防幻觉 + 人工确认机制 + 型号白名单 |
| R2: 角色混淆 | 销售直接确认方案跳过售前 | API层强制角色校验 + 方案状态必须售前确认 |
| R3: 数据安全 | 跨组织数据泄露 | RLS严格隔离 + 审计日志 |
| R4: DeepSeek API不稳定 | 服务中断 | 友好错误处理 + 对话草稿不丢失 + 考虑备用模型 |
| R5: 组织数据迁移意愿低 | 代理商不愿把技术文档上传到云端 | 提供试用期 + 数据导出能力 + 明确数据所有权(属于组织) |
| C1: 价格数据敏感 | 代理商成本和售价是核心机密 | MVP不做价格，只做产品选型 |
| C2: 采购角色延后 | 缺少采购反馈可能导致方案不可执行 | 方案标注"选型建议，采购可行性请人工确认" |

```
├── 确保产品数据准确且最新（型号、参数、停产信息）
├── 确保知识库可检索（销售搜得到、问得准）
├── 建立标准化的方案模板库
├── 每月跟随厂商更新产品数据
└── 监控知识库使用情况（哪些文档被引用最多？哪些搜不到？）

日常任务：
├── 收到施耐德/信捷新产品通知 → 录入产品库
├── 厂商技术文档更新 → 上传知识库（自动解析+Embedding）
├── 售前提交了好方案 → 脱敏后沉淀到知识库作为模板
├── 发现产品参数有误 → 修正
├── 定期用销售的口吻搜索知识库 → 测试检索效果
└── 月底给王总做知识库使用报告

痛点：
├── 🔴 产品数据录入量大：一个系列50+型号，手动录入效率低
├── 🔴 文档版本管理：旧版本还在流传，新版本不知道有没有覆盖
├── 🟡 标签体系难建设：怎么打标签让销售能搜到？
└── 🟡 不知道知识缺口：上传了很多文档，不知道哪些查不到

AI/系统使用场景：
├── 场景A：上传施耐德ATV320产品手册PDF → 系统自动分块 + Embedding → RAG可用
├── 场景B：Excel批量导入新产品型号 → 系统校验格式 → 人工确认
├── 场景C：查看知识库使用统计 → 哪些文档被引用最多 → 优先更新
├── 场景D：搜索测试 → 模拟销售提问 → 检查检索效果 → 调整标签
└── 场景E：方案沉淀审核 → 售前提交的方案 → 审核内容质量 → 批准入库
```

## 5. 核心业务流程 — 跨角色协作

> ⚠️ 关键设计原则：以下所有流程都不是"一个人用AI"，而是**组织内多角色协同完成一个业务闭环**。

### 5.1 流程一：客户需求 → 销售 → 售前（协同选型）

```
这是代理商最核心的业务链路。展示 sales_rep → presales_engineer 的协作模式。

Step 1: 销售收到客户需求（sales_rep）
  "客户来电：新建一个小型污水处理站，3台7.5kW水泵需要控制，
   要求恒压供水，现场环境潮湿有粉尘，预算8万以内，品牌优先级施耐德>信捷。"
  痛点：销售只会记口头需求，说不清技术参数。

Step 2: 销售在Copilot自助初步选型（sales_rep）
  张伟打开 Industrial Copilot，新建对话。
  输入："我需要控制3台7.5kW水泵，恒压供水，现场潮湿有粉尘，
        需要PLC+变频器+触摸屏+压力传感器，预算8万以内，品牌优先施耐德"
  AI返回：
  ├── 推荐PLC: 施耐德M221 TM221CE40R（示例）
  ├── 推荐VFD: 施耐德ATV320U75N4B × 3（示例）
  ├── 推荐HMI: 施耐德HMIGTO5310（示例）
  ├── 推荐传感器: 施耐德XMLP压力传感器（示例）
  ├── 附加建议: IP54控制柜
  └── 注意事项: ATV320 IP20需加防护罩
  销售初步判断合理 → 下一步。

Step 3: 销售发起售前协同审核（sales_rep → presales_engineer）
  张伟点击"转售前审核"按钮，填写备注：
  "李工，污水处理项目，客户预算紧但要施耐德品牌，帮忙看下选型是否合理。
   客户明天要方案。"

  系统行为：
  ├── 对话的 assigned_to 更新为李工
  ├── 李工收到通知："张伟提交了一个选型审核请求"
  └── 对话中标记"已转售前审核"状态

Step 4: 售前审核选型（presales_engineer）
  李工打开对话，看到张伟的备注 + AI推荐方案 + 对话历史。
  快速review：
  ├── ✅ PLC M221: 40点I/O合理，支持Modbus
  ├── ⚠️ VFD ATV320: IP20需要加防护罩，客户现场潮湿有粉尘，建议确认防护方案
  ├── ✅ HMI: 合理
  ├── ⚠️ 传感器: 需要确认量程是否匹配（0-1MPa? 0-1.6MPa?）
  └── 💡 建议: 加1个24V直流电源（ABL8RPS24030），M221 CPU需要

  李工在对话中回复：
  "张伟，总体选型合理。建议：1) ATV320需配IP54防护罩或改用ATV630(IP55)
   2) 补充24V电源型号ABL8RPS24030 3)压力传感器需确认量程。
   确认后我直接出方案。"

Step 5: 销售确认 + 售前出方案
  张伟看到李工的审核 → 跟客户确认压力传感器量程 → 回复李工"量程1.0MPa"
  李工点击"生成方案" → AI生成方案初稿 → 李工补充技术细节 → 标记"已确认"
  → 方案进入方案中心，状态=confirmed

Step 6: 方案交付客户
  李工导出Word → 张伟转发客户
  方案头部标注："✅ 售前已确认 | AI辅助生成"

关键设计点：
  - 销售不必等售前从零开始，AI已做完80%
  - 售前从"执行者"变成"审核者"，效率翻倍
  - 需求在系统中结构化传递，不再靠口头/微信
  - 全程可追溯（谁在什么时候做了什么）
```

### 5.2 流程二：技术方案确认流程（售前主导 + 总监审批）

```
Step 1: 触发方案确认（presales_engineer）
  前置条件：售前李工在Copilot中确认了选型，AI生成了方案初稿
  李工在线编辑方案 → 补充技术细节 → 点击"提交确认"

Step 2: 方案状态判断
  系统根据方案涉及金额或项目风险标记：
  ├── 常规项目(<50万) → 售前自确认 → 直接到"已确认"状态
  └── 大项目(≥50万) → 需要 sales_manager 审批 → 进入审批流程

  （MVP阶段：所有方案售前可自确认，审批流在V1.1实现）

Step 3: 售前确认方案（MVP）
  李工确认：
  ├── 产品清单完整
  ├── 技术参数正确
  ├── 系统架构合理
  ├── 安装注意事项完整
  └── 标注"已确认" → 方案状态=draft→confirmed

Step 4: 方案可导出
  状态=confirmed的方案 → 可导出Word
  导出时：页眉显示"已确认 | 售前工程师: 李工 | 日期: 2026-06-07"

Step 5 (V1.1): 总监审批
  大项目方案 → 进入审批中心 → 王总收到审批通知
  ├── 通过 → 方案可导出
  ├── 驳回 + 批注 → 李工收到驳回通知 + 修改建议 → 重新提交
  └── 超时自动提醒（24h未审批）

Step 6: 方案沉淀（可选）
  已确认+已交付客户的方案 → 售前点击"沉淀到知识库"
  → knowledge_admin审核内容质量 → 通过 → 分块+Embedding → RAG可检索
```

### 5.3 流程三：采购可行性校验流程（V1.2 预留设计）

```
这是跨部门协同的典型场景。MVP阶段不做，但数据库和API需预留扩展点。

预期流程（V1.2）：
Step 1: 方案确认后 → 自动通知procurement角色
Step 2: 采购赵姐查看方案型号清单 → 逐个标记:
  ├── 货期状态: 现货 / 4周 / 12周 / 停产
  ├── 价格区间: 成本范围（销售不可见）
  └── 可行性: ✅可行 / ⚠️需确认 / ❌不可行
Step 3: 采购标记不可行的型号 → 系统自动推荐替代型号（AI匹配）
Step 4: 售前收到采购反馈 → 如需调整 → 修改方案 → 重新确认
Step 5: 全部通过 → 方案标记"采购已确认"

MVP阶段替代方案:
  - 方案导出后，线下发给采购
  - 采购线下反馈
  - 如需调整，售前在线修改方案
```

### 5.4 流程四：知识库更新流程（反馈闭环）

```
【流程A：文档上传（knowledge_admin / presales_engineer）】
Step A1: 上传文档
  小陈收到施耐德最新ATV320产品手册V3.2 PDF
  → 打开知识库 → 上传 → 填写标题、品牌、标签['变频器','ATV320','施耐德']
  
Step A2: 自动处理
  → PDF解析 → 文本分块(chunk_size=500tokens, overlap=50)
  → 批量生成Embedding → 存入knowledge_chunks
  → 文档状态: uploading → processing → ready

Step A3: 验证
  小陈在Copilot中测试搜索"ATV320过载能力" → 确认能检索到V3.2内容
  → 文档正式生效

【流程B：方案沉淀（presales_engineer → knowledge_admin）】
Step B1: 方案提交
  李工完成了一个优秀的水泵控制方案 → 点击"沉淀到知识库"
  → 填写沉淀说明："三泵恒压供水经典方案，含M221+ATV320配置"

Step B2: 管理员审核
  小陈收到沉淀申请 → 审核：
  ├── 方案内容完整度 ✅
  ├── 客户信息已脱敏 ✅
  ├── 技术方案可复用 ✅
  └── 批准入库

Step B3: 入库
  方案内容 → 按章节分块 → 生成Embedding → 存入knowledge_chunks
  → source_type='solution' → 关联原solution_id

【流程C：反馈驱动更新（闭环）】
Step C1: 用户反馈
  销售张伟在AI回答下点击"不准确"
  → 反馈："ATV320的通讯距离实际只有500米，AI说的是1200米"
  
Step C2: 问题追踪
  → 反馈记录进入 review_queue
  → knowledge_admin小陈收到通知

Step C3: 源头修复
  小陈检查对应chunks → 发现引用的是旧版手册
  → 上传新版手册 → 旧文档标记为"已过期" → RAG检索时降权

Step C4: 闭环确认
  → 张伟收到通知："ATV320通讯参数已更新" → 闭环完成
```
